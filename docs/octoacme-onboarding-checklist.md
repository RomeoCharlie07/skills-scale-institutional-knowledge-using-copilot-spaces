## Purpose
Accelerate new team member productivity by providing a structured onboarding experience. This checklist ensures all team members have the context, access, and relationships needed to contribute effectively.

## When to Use
- New hire joining the team
- Internal transfer joining from another team
- Contractor or vendor joining the project
- Returning team member after extended leave

---

## Week 1: Foundation

### Day 1-2: Access & Environment Setup
- [ ] Receive welcome email with onboarding resources
- [ ] Complete HR and compliance onboarding requirements
- [ ] Obtain access to required systems:
  - [ ] Email and calendar
  - [ ] GitHub organization and repositories
  - [ ] Project management tools (GitHub Projects, Jira, etc.)
  - [ ] Communication tools (Slack, Teams, etc.)
  - [ ] Documentation platforms (Confluence, Notion, etc.)
- [ ] Set up local development environment (if applicable)
- [ ] Configure two-factor authentication on all systems

### Day 3-4: Orientation & Context
- [ ] Meet with manager for role expectations and goals discussion
- [ ] Review OctoAcme project management documentation:
  - [ ] [Project Management Overview](./octoacme-project-management-overview.md)
  - [ ] [Roles & Personas](./octoacme-roles-and-personas.md)
  - [ ] [Execution & Tracking](./octoacme-execution-and-tracking.md)
- [ ] Attend team introduction meeting
- [ ] Schedule 1:1s with key team members
- [ ] Review current project status and roadmap

### Day 5: First Contributions
- [ ] Shadow a team standup or sync meeting
- [ ] Review the project board and understand workflow
- [ ] Identify first task or learning assignment
- [ ] Ask questions and document learnings

---

## Week 2-4: Building Competence

### Understanding the Product
- [ ] Review product documentation and user guides
- [ ] Understand the customer personas and use cases
- [ ] Walk through key user flows in the product
- [ ] Review recent releases and roadmap priorities

### Understanding the Process
- [ ] Review remaining process documentation:
  - [ ] [Project Initiation](./octoacme-project-initiation.md)
  - [ ] [Project Planning](./octoacme-project-planning.md)
  - [ ] [Release & Deployment](./octoacme-release-and-deployment.md)
  - [ ] [Retrospective & Continuous Improvement](./octoacme-retrospective-and-continuous-improvement.md)
  - [ ] [Risk Management & Communication](./octoacme-risks-and-communication.md)
- [ ] Understand the Definition of Done and quality expectations
- [ ] Learn the PR workflow and code review process
- [ ] Participate in a sprint planning session

### Building Relationships
- [ ] Complete scheduled 1:1s with team members
- [ ] Identify key stakeholders and their interests
- [ ] Understand escalation paths and decision-making processes
- [ ] Join relevant Slack channels and email distribution lists

### Making Contributions
- [ ] Complete first assigned task
- [ ] Submit first PR (if applicable) and participate in code review
- [ ] Contribute to a team discussion or decision
- [ ] Attend first retrospective

---

## Month 2-3: Building Autonomy

### Deepening Knowledge
- [ ] Develop expertise in assigned focus areas
- [ ] Understand the full system architecture
- [ ] Learn about integrations and dependencies
- [ ] Review historical decisions and their context

### Increasing Responsibility
- [ ] Take ownership of work items independently
- [ ] Participate actively in planning and estimation
- [ ] Contribute to design discussions
- [ ] Provide feedback in retrospectives

### Expanding Impact
- [ ] Identify process improvements or documentation gaps
- [ ] Mentor newer team members (if applicable)
- [ ] Build relationships with cross-functional partners
- [ ] Share knowledge in team forums or demos

---

## Role-Specific Onboarding

### Project Manager
- [ ] Meet with current/previous PM for handoff
- [ ] Review active project plans and timelines
- [ ] Understand stakeholder landscape and communication needs
- [ ] Review risk register and mitigation plans
- [ ] Shadow key meetings before taking ownership

### Product Manager
- [ ] Review product strategy and roadmap
- [ ] Understand user research and feedback channels
- [ ] Meet with key stakeholders (sales, support, engineering)
- [ ] Review backlog and prioritization rationale
- [ ] Understand success metrics and dashboards

### Developer
- [ ] Complete local environment setup
- [ ] Review codebase architecture and conventions
- [ ] Complete a small, well-defined task as first PR
- [ ] Shadow code reviews before leading them
- [ ] Understand testing requirements and CI/CD pipeline

### Scrum Master
- [ ] Observe current team rituals before facilitating
- [ ] Review team agreements and working practices
- [ ] Understand current improvement initiatives
- [ ] Meet with each team member individually
- [ ] Review velocity and capacity trends

### Business Analyst
- [ ] Review current requirements documentation
- [ ] Understand stakeholder landscape
- [ ] Shadow requirement gathering sessions
- [ ] Review in-flight features and their requirements
- [ ] Learn documentation standards and tools

### UX Designer
- [ ] Review design system and style guides
- [ ] Understand user research findings and personas
- [ ] Review current designs and design debt
- [ ] Set up design tools and asset libraries
- [ ] Meet with Product and Engineering leads

### Technical Writer
- [ ] Review existing documentation inventory
- [ ] Understand publishing workflows and tools
- [ ] Review style guides and templates
- [ ] Identify priority documentation needs
- [ ] Meet with subject matter experts

### QA Lead
- [ ] Review test strategy and coverage reports
- [ ] Understand test environments and automation
- [ ] Review defect backlog and quality metrics
- [ ] Meet with development leads
- [ ] Set up testing tools and frameworks

---

## Manager Responsibilities

### Before Start Date
- [ ] Prepare access requests and equipment
- [ ] Create onboarding schedule and meeting invites
- [ ] Identify onboarding buddy or mentor
- [ ] Prepare welcome materials and first assignments

### First Week
- [ ] Conduct role expectations meeting
- [ ] Introduce to team and key stakeholders
- [ ] Daily check-ins to address questions
- [ ] Provide context on current projects

### First Month
- [ ] Weekly 1:1s focused on onboarding progress
- [ ] Provide regular feedback on initial work
- [ ] Address any access or resource issues
- [ ] Adjust onboarding plan based on progress

### End of Onboarding (90 days)
- [ ] Conduct formal onboarding review
- [ ] Gather feedback on onboarding experience
- [ ] Set performance goals for next period
- [ ] Confirm full integration into team

---

## Onboarding Feedback

At the end of your onboarding, please share feedback to help us improve:
- What worked well during your onboarding?
- What was missing or unclear?
- What would you recommend we change?
- How prepared do you feel to perform your role?

Share feedback with your manager or submit to the onboarding improvement backlog.
